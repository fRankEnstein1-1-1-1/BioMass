## SECTION 1: PROJECT OVERVIEW

**1. Project Purpose**
BioMass is a full-stack application designed to process and analyze drone imagery to automatically estimate vegetation biomass and carbon storage. 

**2. Main Problem Being Solved**
Manual calculation of biomass from drone imagery is tedious. This system automates the generation of Digital Surface Models (DSM) and Digital Terrain Models (DTM) via WebODM, computes a Canopy Height Model (CHM), and leverages these to estimate biomass and carbon storage.

**3. Target Users**
Environmentalists, surveyors, agricultural engineers, and researchers needing automated drone imagery analysis.

**4. User Roles**
Currently, only a single standard user role is implemented. Users can sign up, log in, create projects, and upload images. There is no administrative tier.

**5. Main Features**
- User Authentication (Signup/Login)
- Project Creation
- Bulk Drone Image Upload (Drag-and-drop)
- Integration with WebODM for orthophoto, DSM, and DTM generation
- GeoTIFF processing to generate Canopy Height Models (CHM)
- Environmental calculations (Canopy area, max/mean height, biomass, carbon estimates)

**6. Business Workflow**
1. User creates an account and logs in.
2. User creates a "New Project" and uploads a batch of drone images.
3. The frontend creates the project via API, then uploads the images.
4. Backend acts as a bridge, creating a corresponding project and task in WebODM.
5. WebODM processes the images.
6. Backend downloads the resulting DSM and DTM GeoTIFFs.
7. Backend analyzes the GeoTIFFs to calculate biomass and carbon.
8. (Pending Frontend) User views the results.

**7. High-Level Architecture**
- **Frontend**: React.js SPA built with Vite and TailwindCSS.
- **Backend**: Node.js/Express.js RESTful API.
- **Database**: MongoDB for user and project metadata.
- **Processing Engine**: WebODM (OpenDroneMap) running in Docker.

**8. Current Implementation Status**
- **Backend**: ~85% complete. The core integration with WebODM, image uploads, and complex GeoTIFF mathematics are implemented.
- **Frontend**: ~40% complete. Auth pages and the upload page exist. The project listing and results visualization pages are entirely missing.
- **Security**: ~10% complete. Passwords are in plain text, and APIs lack authorization barriers.

**9. Estimated Project Completion Percentage**
Approximately **65%** overall. The hardest parts (WebODM integration and GeoTIFF manipulation) are done, but the frontend needs significant work to make the app usable.

## SECTION 5: API DOCUMENTATION

> [!WARNING]
> None of the backend APIs currently require Authentication headers (JWT). This is a critical security flaw.

### Authentication Endpoints
1. **POST `/api/auth/signup`**
   - **Body**: `{ name, email, password }`
   - **Purpose**: Creates a new user.

2. **POST `/api/auth/login`**
   - **Body**: `{ email, password }`
   - **Purpose**: Authenticates a user (checks plain text password).

### Project Endpoints
3. **POST `/api/projects`**
   - **Body**: `{ projectName, userId }`
   - **Purpose**: Initializes a new project in the DB.

4. **POST `/api/upload/:projectId`**
   - **Body**: `multipart/form-data` containing `images` (array)
   - **Purpose**: Uploads images to Node, forwards them to WebODM, and starts a WebODM task.

5. **GET `/api/projects/:projectId/status`**
   - **Purpose**: Fetches WebODM task status and updates DB.

6. **POST `/api/projects/:projectId/download-assets`**
   - **Purpose**: Downloads the DSM and DTM files from WebODM locally.

7. **GET `/api/projects/:projectId/generate-chm`**
   - **Purpose**: Generates a Canopy Height Model from DSM and DTM.

8. **POST `/api/projects/:projectId/analyze`**
   - **Purpose**: Runs the full suite of calculations (Canopy, Biomass, Carbon) and saves to DB.

**Current Status: CRITICAL RISK**

- **Login Flow**: Frontend sends email/password. Backend finds user and compares passwords using `===` (plain text comparison). It returns the user object.
- **Session Flow**: Frontend saves the raw user object in `localStorage` and uses it to determine if the user is logged in. 
- **JWT / Tokens**: **Not implemented.**
- **Password Hashing**: **Not implemented.**
- **Route Protection**: React Router prevents accessing `/` if local state is null. Backend routes are completely unprotected. Anyone can access any API endpoint if they know a `projectId`.

## SECTION 10: USER WORKFLOWS

**1. Create Item (Upload Images):**
- **Action**: User drags images into dropzone and clicks "Upload".
- **Frontend**: Calls `createProject` API, then loops images into FormData and calls `uploadImages`.
- **Backend**: Saves files via Multer locally -> Calls WebODM API to create project -> Pushes images to WebODM task -> Returns success.

**2. Missing Workflows (Critical Gap):**
Currently, after upload, the frontend shows "Upload Complete". It **does not** poll the backend to check WebODM status, nor does it automatically trigger the analysis steps. 
The backend has routes to download assets and run analysis (`/analyze`), but the frontend never calls them.

## SECTION 20: EXECUTIVE SUMMARY & SURVIVAL GUIDE

**Current Project Status**: The project is a solid prototype with the hardest technical challenges (WebODM integration and manual GeoTIFF raster parsing in JavaScript) already solved. However, the application is currently "disconnected"; the frontend can upload data, but cannot retrieve or view the processed results.

### Frontend Developer Survival Guide
Welcome to BioMass! To get started immediately:

1. **Start Services**:
   - WebODM: `cd WebODM && ./webodm.sh start`
   - Backend: `cd Backend && npm run start` (Requires `.env` with `MONGO_URL` and WebODM credentials)
   - Frontend: `cd frontend && npm run dev`

2. **The "Missing Link"**:
   Read `Backend/Controllers/projectController.js`. Notice endpoints like `getProjectStatus`, `downloadProjectAssets`, and `runEnvironmentalAnalysis`. 
   **Your main job is to call these endpoints from the frontend.**
   After a user uploads images, you need to ping `getProjectStatus` until it says "completed", then hit `download-assets`, then hit `analyze`. 

3. **Database Gotcha**:
   If you manually test logins, remember passwords are in plain text right now. Don't use your real personal passwords.

4. **UI Styling**:
   Stick to Tailwind CSS. The app uses a dark theme (`bg-slate-900`) with blue highlights. Check `LoginPage.jsx` for examples of the glassmorphism card styles (`glass-card` class in `index.css`).
